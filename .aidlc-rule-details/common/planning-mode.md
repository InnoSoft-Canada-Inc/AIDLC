# Planning-Only Mode

## Purpose

Allow users to plan a feature or task WITHOUT proceeding to Construction. The planning output is saved to the user's docs folder, creating a reusable planning document that can later serve as an anchor doc for implementation.

**Use case**: User wants to think through a feature thoroughly before committing to build it.

---

## Trigger Phrase

```
Using AI-DLC, plan [feature/task description]
```

**Key distinction**: Uses "plan" instead of "implement" or generic task description.

**Examples**:
- `Using AI-DLC, plan user authentication with OAuth`
- `Using AI-DLC, plan the shopping cart feature`
- `Using AI-DLC, plan API rate limiting system`
- `Using AI-DLC, plan database migration to PostgreSQL`

---

## Workflow

### Phase 1: Track Selection

When "plan" trigger is detected:

1. **Recognize planning mode**: Detect "plan" keyword in trigger phrase
2. **Announce mode**:
   ```
   Selected: Planning Mode — You want to plan this feature before implementation.
   Output will be saved to your docs folder. Construction will NOT begin.
   ```
3. **Wait for confirmation** before proceeding

### Phase 2: Inception (Full Execution)

Execute all applicable Inception stages:

| Stage | Executes? | Notes |
|-------|-----------|-------|
| **Workspace Detection** | ✅ Always | Load project context, guidelines |
| **Reverse Engineering** | ✅ If brownfield, no prior artifacts | Understand existing system |
| **Requirements Analysis** | ✅ Always | Capture what user wants to build |
| **User Stories** | ✅ If applicable | User-facing features need stories |
| **Workflow Planning** | ✅ Always | Determine how it would be built |
| **Application Design** | ✅ If applicable | New components, architecture decisions |
| **Units Generation** | ✅ If complex | Break down into units if needed |

**Note**: All stages run normally with approval gates. The only difference is what happens after Inception completes.

### Phase 3: Generate Planning Document

After Inception completes, generate a consolidated planning document:

**Output location**: `{knowledge_base}/planning/{feature-slug}-plan.md`

Example: `docs/planning/oauth-authentication-plan.md`

**Document structure**:

```markdown

**Generated**: {timestamp}

**Domain**: {proposed domain}

To implement this feature, run:
```


## Using a Planning Document Later

When user wants to implement a planned feature:

### Option 1: Reference in Trigger

```
Using AI-DLC, implement OAuth authentication using docs/planning/oauth-plan.md
```

**Behavior**:
1. Load the planning document as anchor
2. Skip redundant Inception stages (requirements already captured)
3. Validate planning doc is still current
4. Proceed to Construction

### Option 2: Automatic Detection

```
Using AI-DLC, implement OAuth authentication
```

**Behavior**:
1. During Workspace Detection, check for matching planning doc
2. If found: "I found a planning document for this feature. Use it? [Yes/No]"
3. If yes: Load as anchor, skip redundant Inception
4. If no: Run full Inception (may override previous planning)

### Skipping Redundant Stages

When using a planning document as anchor:

| Stage | Skip? | Condition |
|-------|-------|-----------|
| Requirements Analysis | ✅ Skip | If planning doc has requirements |
| User Stories | ✅ Skip | If planning doc has stories |
| Workflow Planning | ⚠️ Partial | Validate recommendations, may adjust |
| Application Design | ✅ Skip | If planning doc has design |
| Units Generation | ✅ Skip | If planning doc has units |

**Always run**: Workspace Detection (for current context)

## Planning Document Lifecycle

### States

| State | Meaning |
|-------|---------|
| 📋 Planning Complete | Ready for implementation |
| 🔄 Implementation Started | Construction has begun using this plan |
| ✅ Implemented | Feature completed, plan archived |
| ⚠️ Outdated | Project changes may have invalidated this plan |
| ❌ Superseded | Newer plan exists for this feature |

### Updating a Plan

```
Using AI-DLC, update plan for OAuth authentication
```

**Behavior**:
1. Load existing planning doc
2. Ask what needs to change
3. Re-run affected Inception stages
4. Update planning doc
5. Mark as updated with new timestamp

### Archiving a Plan

When a planned feature is implemented:
1. Move planning doc to `{knowledge_base}/planning/archive/`
2. Update status to "✅ Implemented"
3. Link to the actual feature documentation

## Error Handling

| Scenario | Handling |
|----------|----------|
| **User says "implement" mid-planning** | Ask: "Did you want to switch to implementation mode? This will continue to Construction." |
| **Planning doc already exists** | Ask: "A planning document already exists. Update it or create new?" |
| **Knowledge base not configured** | Save to `aidlc-docs/_shared/planning/` as fallback |
| **Planning doc referenced but not found** | Warn user, offer to run full Inception |

---

## Integration with Tracks

Planning mode is **track-agnostic** — it always runs Inception fully regardless of what track would be selected for implementation.

When the plan is later used for implementation:
- Track selection happens at implementation time
- Plan may recommend a track, but user can override
- Complexity assessment from planning informs track selection
